# CLAUDE.md

This file orients a new Claude Code session on this project. Read this first, then `PLAN.md` for full technical detail on whatever section is relevant to the current task.

## What this is

A portfolio project for a Data Scientist CV: **LLM-Augmented Transaction Anomaly Detection Engine**. A synthetic retail-banking transaction dataset (1.2M rows, generated by us — no such labeled public dataset exists), an Isolation Forest / LightGBM anomaly-scoring pipeline, a Claude reasoning layer that explains flagged transactions for investigator triage, statistically validated decision thresholds, model-risk documentation styled on SR 11-7 / PRA SS1/23, and a deployed demo with a clickable CV link.

Full plan, architecture, rationale, and week-by-week roadmap: **`PLAN.md`** (16 sections, table of contents at top). Don't duplicate that content here — this file is state, not spec.

## Binding decisions (do not relitigate without asking)

- **LLM provider is Claude**, not GPT-4/OpenAI, despite the original CV draft saying GPT-4. Uses the Anthropic API (`console.anthropic.com`, pay-as-you-go — separate from Claude.ai Pro, which does NOT include API access). Haiku for bulk explanation generation, Sonnet only for a small sample used to measure typology-classification accuracy.
- **The LLM backend is swappable** between the Anthropic API and a locally-hosted model (Ollama, via its OpenAI-compatible `/v1` endpoint) with zero script changes — a one-line `LLM_PROVIDER` env var flip. `llm/client.py`'s `get_llm_client()` factory is the only place that branches; every caller (`generate_explanations.py`, later `api/`) codes against the `LLMClient` interface. Anthropic remains the primary/default path; the local swap is a tested alternative requested for flexibility, not a scope change to the project's LLM story.
- **Stack is Streamlit + FastAPI + Docker Compose on a single AWS EC2 instance.** No Next.js, no Vercel, no Render — user doesn't know them and they're not needed for a Data Scientist role. Postgres is self-hosted in a Docker container on the same box, not RDS (RDS flagged as too costly for this).
- **Data is synthetic, generated by us**, with injected AML typologies (structuring, layering, round-amount anomalies, velocity spikes, peer-group deviation, geographic risk) mapped to FATF red-flag categories. Ground truth is withheld from the feature set and used only for evaluation + a "hand-labelled validation set" stand-in.
- **Integrity rule:** the CV bullet numbers (65% effort reduction, 71%→89% precision, 40% fewer false positives) are placeholders/target shape only. Real numbers get written down in Week 8 from what the pipeline actually measures — never backfilled to match the draft. See `PLAN.md` §00 and §15.
- **Framing rule:** this is an independent portfolio project on synthetic data, not a real bank engagement or actual regulatory validation. The model-risk report says so on page one; say the same out loud in interviews.

Root cause identified, not just a flaky model: `OpenAICompatibleClient` targets Ollama's OpenAI-compatible `/v1/chat/completions` shim (chosen deliberately so the same class also works against vLLM/LM Studio with zero code changes), and that shim only supports the generic `response_format={"type": "json_object"}` mode (valid-JSON guarantee) - not OpenAI's `json_schema` mode (schema-conformance guarantee). <summary>Finding: threshold tuning did not beat the naive capacity cutoff (click for the full story)</summary>

Two attempts, same qualitative result:

1. **First attempt** — selected the threshold on a single VALIDATION block (the 15% of data immediately before TEST). Result: precision 67.0% @ capacity on TEST, but the z-test-chosen "improved" threshold made FP rate *worse* on TEST (+17.7%), despite looking like a significant improvement on VALIDATION (p=0.0001). Classic single-split overfitting risk.
2. **Second attempt** (current code) — refit the tuned model across 4 expanding-window folds *within* TRAIN and pooled out-of-fold predictions (512K rows spanning multiple time windows) for threshold selection instead. More statistically robust by construction, but produced a *lower* capacity-threshold precision on TEST (58.3%, not 67.0%) and the z-test-chosen threshold again made FP rate worse (+17.3%), not better.

Checked whether this is target drift (anomaly rate or typology mix shifting across train/val/test): **no** - anomaly rate is 1.52%/1.68%/1.55% and typology proportions are within a few points of each other across all three splits. So the gap is either (a) score-calibration drift in the model itself over time, or (b) finite-sample noise in the threshold estimate - not distinguished yet, and not chased further this session per the integrity rule (two consistent negative results is a real finding, not a reason to keep searching for a threshold that "works").

**Decision:** report 58.3% (the more statistically defensible, larger-sample OOF-calibrated number) as the honest "after" precision, not 67.0% (which looked better but came from a single, smaller, less robust calibration window). Report the threshold-tuning attempt as a validated null result, not a success - the two-proportion z-test methodology worked exactly as intended: it caught that the "improvement" wasn't real, on both attempts. Revisit in Week 5 (`model_validation_report.md` sensitivity analysis) if there's time - candidate follow-ups: cost-weighted optimization instead of FP-rate-only, or checking per-typology threshold sensitivity rather than one global threshold.
